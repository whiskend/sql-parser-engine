## 2. 구조를 어떻게 잡았는가

### 참고하는 README 섹션

* `2. 구조를 어떻게 잡았는가`

### 발표 멘트

저희는 SQL을 문자열로 처리하지 않고, `Token -> AST -> 실행 결과`로 바뀌는 흐름으로 정리했습니다.

그래서 구조도 자연스럽게 `Lexer`, `Parser`, `Executor`, `Storage`로 나뉘었습니다.  
Lexer는 자르고, Parser는 데이터 구조를 만들고, Executor는 실행으로 연결하고, Storage는 파일에 반영합니다.

---

## 3. 설계 선택 1: 저장 포맷을 어떻게 볼 것인가

### 참고하는 README 섹션

* `3. 설계 선택 1: 저장 포맷을 어떻게 볼 것인가`

### 발표 멘트

저장 포맷은 자유였기 때문에, 여기서 첫 번째 설계 선택이 나왔습니다.

처음에는 CSV와 binary를 고민했지만, binary는 읽기 어렵고 CSV도 escape 처리가 번거로워 디버깅에 적합하지 않다고 판단했습니다.

그래서 저희는 성능보다 디버깅을 우선하여 `|` 구분자를 쓰는 텍스트 포맷을 선택했습니다.  

그러니까 학습용 미니 프로젝트에서 먼저 필요한 건 효울적인 포맷보다는, 개발자가 상태를 쉽게 읽고 검증할 수 있는 형식이라고 판단했습니다.

## 4. 설계 선택 2: 입력 처리 문제와 개선

### 참고하는 README 섹션

* `4. 설계 선택 2: 입력 처리 문제와 개선`

### 발표 멘트

여기서부터는 구현하면서 겪었던 부분을 말씀드리겠습니다.

초기 구현에서는 SQL 파일이 두 줄 이상일 때 제대로 동작하지 않는 문제가 있었습니다.

그래서 입력을 한 줄 문자열이 아니라, 세미콜론 기준으로 나눠 순차적으로 실행하도록 수정했습니다.

또 구조를 확인하기 위해 WHERE 조건을 간단히 추가해, Parser와 Executor 흐름이 자연스럽게 이어지는지도 확인했습니다.

## 5. 설계 선택 3: 파일 기반 DB에도 최소한의 원칙은 필요하다고 봤다

### 참고하는 README 섹션

* `5. 설계 선택 3: 파일 기반 DB에도 최소한의 원칙은 필요하다고 봤다`

### 발표 멘트

다음 판단은 데이터 무결성이었습니다.

요구사항에는 key 제약이 없지만, `id`가 계속 중복되면 파일 기반이라도 데이터가 금방 모호해집니다.  
그래서 저희는 완전한 제약 시스템까지는 아니더라도, 최소한 `id` 중복은 막는 것이 맞다고 봤습니다.

즉, 파일 기반 DB라도 "최소한의 무결성은 지켜 보자"는 판단이었습니다.

그리고 또 스키마 파일이 바뀌어도 동작할 수 있도록 처리했습니다.
