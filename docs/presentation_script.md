## 2. 구조를 어떻게 잡았는가

### 참고하는 README 섹션

* `4. Top-Down 관점의 전체 흐름`
* `5. 모듈 구조`
* `6. 함수 호출 흐름`

### 발표 멘트

저희는 SQL을 문자열로 처리하지 않고, `Token -> AST -> 실행 결과`로 바뀌는 흐름으로 정리했습니다.

그래서 구조도 자연스럽게 `Lexer`, `Parser`, `Executor`, `Storage`로 나뉘었습니다.  
Lexer는 자르고, Parser는 구조를 만들고, Executor는 실행으로 연결하고, Storage는 파일에 반영합니다.

## 3. 설계 선택 1: 저장 포맷을 어떻게 볼 것인가

### 참고하는 README 섹션

* `7. 저장 구조`

### 발표 멘트

저장 포맷은 자유였기 때문에, 여기서 첫 번째 설계 선택이 나왔습니다.

처음에는 CSV와 binary를 둘 다 고민했습니다.  
binary는 빠를 수 있지만 사람이 직접 읽기 어렵고, CSV도 쉼표와 escape를 계속 신경 써야 해서 디버깅과 테스트에 아주 편한 포맷은 아니라고 봤습니다.

그래서 저희는 `|` 구분자를 쓰는 텍스트 포맷을 선택했습니다.  
이 선택의 의미는 성능보다 "직접 열어서 바로 확인할 수 있는가", "테스트 기대값을 쉽게 비교할 수 있는가"를 우선한 것입니다.

그러니까 미니 프로젝트에서 먼저 필요한 건 복잡한 포맷보다도, 개발자가 상태를 쉽게 읽고 검증할 수 있는 형식이라고 판단했습니다.

## 4. 설계 선택 2: 기본 요구사항보다 현실적인 입력으로 확장

### 참고하는 README 섹션

* `6. 함수 호출 흐름`
* `8. INSERT 흐름`
* `9. SELECT 흐름`

### 발표 멘트

여기부터는 저희가 요구사항을 조금 더 현실적으로 해석했던 부분입니다.  
저희는 기본 기능을 맞추는 데서 멈추기보다, 실제로 SQL 파일을 사용하면 어떤 문제가 먼저 생길지를 같이 봤습니다.

가장 먼저 걸린 건 입력 처리였습니다.  
SQL 파일은 한 줄로만 쓰이지 않고, 줄바꿈이 들어가거나 여러 문장이 한 파일에 들어갈 수 있습니다.  
처음에는 이 부분을 단순하게 볼 수도 있었지만, 그러면 텍스트 파일로 SQL을 받는다는 요구사항이 실제 사용 방식과 조금 멀어진다고 느꼈습니다.  
그래서 입력을 "한 줄 문자열"이 아니라 "토큰 스트림"으로 해석하고, 세미콜론 단위로 순차 실행하는 구조로 바꿨습니다.

이 선택은 편의 기능을 추가했다기보다, 입력 처리의 현실성을 조금 더 챙긴 판단에 가까웠습니다.

또 조회도 단순 `SELECT *`만 있으면 구조 확장성을 보여 주기 어렵다고 생각했습니다.  
그래서 `WHERE column = literal`까지만 얇게 넣어서, Parser가 조건을 AST에 담고 Executor가 실제 row filtering을 하도록 확장했습니다.

여기서 중요했던 건 WHERE 기능 자체보다, Parser -> Executor 구조가 무너지지 않고 자연스럽게 확장되는지 확인하는 것이었습니다.

## 5. 설계 선택 3: 파일 기반 DB에도 최소한의 원칙은 필요하다고 봤다

### 참고하는 README 섹션

* `8. INSERT 흐름`
* `17. 엣지 케이스`
* `18. 한계와 향후 개선점`

### 발표 멘트

다음 판단은 데이터 무결성이었습니다.

요구사항에는 key 제약이 없지만, `id`가 계속 중복되면 파일 기반이라도 데이터가 금방 모호해집니다.  
그래서 저희는 완전한 제약 시스템까지는 아니더라도, 최소한 `id` 중복은 막는 것이 맞다고 봤습니다.

이 선택의 의미는 파일 기반 DB라도 DB의 기본 원칙은 조금 반영해 보자는 것이었습니다.  
즉, "저장만 되면 된다"보다는 "최소한의 무결성은 지켜 보자"는 판단이었습니다.

같은 맥락에서 스키마 파일이 바뀌었을 때도, 시스템이 그 변화를 어느 정도 받아들일 수 있어야 한다고 봤습니다.  
실제로 파일 기반 DB에서는 개발자가 `db` 디렉터리를 직접 수정할 수 있기 때문입니다.

그래서 현재 schema보다 저장 row가 짧으면 부족한 값을 채우고, 길면 잘라서 맞추도록 읽기 경로를 정리했습니다.  
완벽한 마이그레이션은 아니지만, 코드와 데이터 구조를 너무 강하게 묶지 않으려는 선택이었습니다.  
파일 기반 DB의 유연성을 조금이라도 살리고 싶었던 판단이라고 보시면 될 것 같습니다.

## 7. 테스트와 검증

### 참고하는 README 섹션

* `13. 테스트 전략`
* `17. 엣지 케이스`

### 발표 멘트

다음은 테스트입니다.

README의 테스트 전략을 보시면, 저희는 Lexer, Parser, Storage, Executor를 각각 단위 테스트로 검증하고, 마지막에 integration test로 실제 바이너리 실행까지 확인했습니다.

여기서 중요한 건 테스트 개수보다, 어떤 문제를 검증했는가였습니다.  
단순 정상 케이스뿐 아니라, 중복 `id`, WHERE 결과 0건, schema 변경 뒤 row 읽기, 여러 문장 SQL 같은 상황을 같이 봤습니다.

발표 직전 기준 명령은 `make && make test`였고, 이 과정을 통해 추가 구현도 구조 안에서 제대로 동작하는지 확인했습니다.

## 8. 데모

### 참고하는 README 섹션

### 발표 멘트

즉, 입력이 조금 더 현실적으로 바뀌거나 조건이 추가되어도 전체 흐름은 흔들리지 않았습니다.

## 9. 협업 방식

### 참고하는 README 섹션

* `15. 우리가 사용한 협업 방식`
* `16. 처음 읽을 때 추천하는 코드 읽기 순서`

### 발표 멘트

협업 방식도 짧게 말씀드리겠습니다.

저희는 오전에는 Top-Down으로 구조를 먼저 이해하며 각자 codex에게 넣을 통합 작업 지시서를 작성했고, 점심 이후에 작업지시서를 비교해 가장 일관된 안을 골랐습니다.  
그 뒤에 통합 지시서와 각 모듈별 작업 지시서를 만들어 병렬 구현했고, 저녁에는 다시 코드를 읽으면서 발표 가능한 수준으로 정리했습니다.

이 방식이 좋았던 이유는, 구현 속도와 코드 이해를 같이 챙길 수 있었다는 점입니다.
