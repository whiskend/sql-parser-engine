## 2. 구조를 어떻게 잡았는가

### 참고하는 README 섹션

* `2. 구조를 어떻게 잡았는가`

### 발표 멘트

저희는 SQL을 문자열로 처리하지 않고, `Token -> AST -> 실행 결과`로 바뀌는 흐름으로 정리했습니다.

그래서 구조도 자연스럽게 `Lexer`, `Parser`, `Executor`, `Storage`로 나뉘었습니다.  
Lexer는 자르고, Parser는 구조를 만들고, Executor는 실행으로 연결하고, Storage는 파일에 반영합니다.

---

## 3. 설계 선택 1: 저장 포맷을 어떻게 볼 것인가

### 참고하는 README 섹션

* `3. 설계 선택 1: 저장 포맷을 어떻게 볼 것인가`

### 발표 멘트

저장 포맷은 자유였기 때문에, 여기서 첫 번째 설계 선택이 나왔습니다.

처음에는 CSV와 binary를 고민했습니다.  
binary는 빠를 수 있지만 사람이 직접 읽기 어렵고, CSV도 쉼표와 escape를 계속 신경 써야 해서 디버깅과 테스트에 아주 편한 포맷은 아니라고 봤습니다.

그래서 저희는 `|` 구분자를 쓰는 텍스트 포맷을 선택했습니다.  
이 선택의 의미는 성능보다 디버깅을 우선한 것입니다.

그러니까 미니 프로젝트에서 먼저 필요한 건 복잡한 포맷보다는, 개발자가 상태를 쉽게 읽고 검증할 수 있는 형식이라고 판단했습니다.

## 4. 설계 선택 2: 입력 처리 문제와 개선

### 참고하는 README 섹션

* `4. 설계 선택 2: 입력 처리 문제와 개선`

### 발표 멘트

여기서는 구현하면서 겪었던 부분을 하나 말씀드리겠습니다.

처음에 기본 요구사항대로 기능을 구현해봤는데, SQL 파일이 두 줄 이상이 되면 제대로 동작하지 않는 문제가 있었습니다.  
codex로 생성된 초기 코드가 한 줄 입력만 가정하고 있었기 때문입니다.

그래서 이 부분이 실제 사용 방식과 맞지 않다고 판단해서, 여러 줄로 된 SQL도 처리할 수 있도록 수정했습니다.  
입력을 한 줄 문자열로 보지 않고, 세미콜론 기준으로 나눠서 순차적으로 실행하도록 구조를 바꿨습니다.

또 SELECT도 처음에는 단순한 형태만 있었는데, 구조를 조금 더 확인해보고 싶어서 WHERE 조건을 간단하게 추가해봤습니다.  
Parser가 조건을 AST에 담고, Executor가 실제로 filtering을 하는 흐름이 자연스럽게 이어지는지 확인하는 목적이었습니다.

## 5. 설계 선택 3: 파일 기반 DB에도 최소한의 원칙은 필요하다고 봤다

### 참고하는 README 섹션

* `5. 설계 선택 3: 파일 기반 DB에도 최소한의 원칙은 필요하다고 봤다`

### 발표 멘트

다음 판단은 데이터 무결성이었습니다.

요구사항에는 key 제약이 없지만, `id`가 계속 중복되면 파일 기반이라도 데이터가 금방 모호해집니다.  
그래서 저희는 완전한 제약 시스템까지는 아니더라도, 최소한 `id` 중복은 막는 것이 맞다고 봤습니다.

이 선택의 의미는 파일 기반 DB라도 DB의 기본 원칙은 조금 반영해 보자는 것이었습니다.  
즉, "저장만 되면 된다"보다는 "최소한의 무결성은 지켜 보자"는 판단이었습니다.

그리고 또 스키마 파일이 바뀌어도 동작할 수 있도록 처리했습니다.

읽을 때 row 길이가 schema와 다르면, 부족하면 채우고 길면 잘라서 맞추도록 했습니다.

## 6. 시행착오와 구조 변경

### 참고하는 README 섹션

* `6. 시행착오와 구조 변경`

### 발표 멘트

이런 선택들이 처음부터 한 번에 나온 것은 아닙니다.

처음에는 기본 요구사항만 우선 맞춰봤습니다.
그런데 테스트를 붙이고 파일을 직접 만져 보니, 줄바꿈이 들어간 SQL, 중복 `id`, schema 변경 같은 예외 상황이 금방 드러났습니다.

완전한 예외 처리는 아니지만, 실제 입력 상황에서 발생할 수 있는 다양한 예외를 고려해 안정적으로 동작하도록 코드를 조금씩 바꿨습니다.

## 7. 테스트와 검증

### 참고하는 README 섹션

* `7. 테스트와 검증`

### 발표 멘트

다음은 테스트입니다.

README의 테스트와 검증 섹션을 보시면, 저희는 Lexer, Parser, Storage, Executor를 각각 단위 테스트로 검증하고, 마지막에 integration test로 실제 바이너리 실행까지 확인했습니다.

여기서 중요한 건 테스트 개수보다, 어떤 문제를 검증했는가였습니다.  
단순 정상 케이스뿐 아니라, 중복 `id`, WHERE 결과 0건, schema 변경 뒤 row 읽기, 여러 문장 SQL 같은 상황을 같이 봤습니다.

발표 직전 기준 명령은 `make && make test`였고, 이 과정을 통해 추가 구현도 구조 안에서 제대로 동작하는지 확인했습니다.

## 8. 데모

### 참고하는 README 섹션

* `8. 데모`

### 발표 멘트

먼저 기본 `INSERT`와 `SELECT` 흐름을 짧게 보여 드리고,  
그다음 `WHERE`가 들어간 조회와 여러 문장이 들어간 SQL 파일을 한 번에 실행해 보겠습니다.

여기서 보여 드리고 싶은 건 기능 목록이 아니라, 처음에 설명드린 구조가 그대로 확장되어도 유지된다는 점입니다.  
즉, 입력이 조금 더 현실적으로 바뀌거나 조건이 추가되어도 전체 흐름은 흔들리지 않았습니다.

## 9. 협업 방식

### 참고하는 README 섹션

* `9. 협업 방식`

### 발표 멘트

협업 방식도 짧게 말씀드리겠습니다.

저희는 오전에는 Top-Down으로 구조를 먼저 이해하며 각자 codex에게 넣을 통합 작업 지시서를 작성했고, 점심 이후에 작업지시서를 비교해 가장 일관된 안을 골랐습니다.  
그 뒤에 통합 지시서와 각 모듈별 작업 지시서를 만들어 병렬 구현했고, 저녁에는 다시 코드를 읽으면서 발표 가능한 수준으로 정리했습니다.

이 방식이 좋았던 이유는, 구현 속도와 코드 이해를 같이 챙길 수 있었다는 점입니다.
